# ByteDance/TikTok Passport SDK Patterns

## Overview
TikTok-affiliated properties (SoundOn, TikTok Shop, CapCut, etc.) share ByteDance's passport SDK for authentication. Understanding these patterns accelerates testing on any ByteDance-owned target.

## Authentication Architecture

### Passport Endpoints
```
/passport/web/email/send_code/?aid={APP_ID}&account_sdk_source=web&sdk_version=2.1.10-tiktok&language=en
/passport/web/email/register_verify_login/?aid={APP_ID}&...
/passport/web/email/login/?aid={APP_ID}&...
/passport/safe/login_op_log/
```

### Known App IDs
- `aid=5049` — SoundOn (tiktok_for_artists)

### Anti-Bot Stack
- **webmssdk.js** — Fingerprinting + `msToken` generation (silent, no visual captcha)
- **goofy-captcha** — Visual captcha (loaded from `sf16-website-login.neutral.ttwstatic.com/obj/tiktok_web_login_static/goofy-captcha/`)
- **X-Bogus** — Request signature (computed client-side via webmssdk)
- **X-Gnarly** — Additional request signature
- **verifyFp** — Device fingerprint cookie (`verify_xxxxx_yyyy`)

### Rate Limiting
- IP-based server-side (not bypassable via header spoofing)
- Client-side toast via cookie/storage flag (clearable but server still blocks)
- Triggers after ~3 failed registration attempts from same IP
- Cooldown appears to be ~30-60 minutes
- Error message: "Maximum number of attempts reached. Try again later."

### Registration Flow (Email)
1. `POST /passport/web/email/send_code/` — body: `email=X&type=1&mix_mode=1`
2. Code arrives via email from `no-reply@mail.soundon.global`
3. `POST /passport/web/email/register_verify_login/` — body: `email=X&password=Y&code=Z&mix_mode=1`
4. On success: httpOnly `sessionid` + `sessionid_ss` + `sid_guard` cookies set

### Error Codes
- `error_code: 0` — success
- `error_code: 3` — "Enter the correct parameter" (missing msToken/X-Bogus)
- `error_code: 1011` — "Account doesn't exist"
- `error_code: 1204` — "Something went wrong. Try again." (code expired/already used)

### Session Cookies (httpOnly)
- `sessionid` — primary session
- `sessionid_ss` — secondary/secure session
- `sid_guard` — session guard with expiry info
- NOT visible via `document.cookie` — must verify by navigating to authenticated page

## Testing Implications

### Direct API Calls Require:
- Valid `msToken` (generated by webmssdk fingerprinting)
- `X-Bogus` signature (computed per-request)
- `verifyFp` cookie
- Without these → `error_code: 3`

### Browser Automation Works Because:
- webmssdk runs naturally in browser context
- msToken/X-Bogus generated automatically on fetch/XHR
- No visual captcha for email registration (only rate limiting)

### BOLA/IDOR Testing Notes:
- Session is tied to userId (in session server-side)
- artistId parameter is accepted on many endpoints but server uses session's userId
- Need actual second account with data to prove IDOR (empty accounts show empty regardless)
- Revenue/analytics endpoints: return `errorCode: 0` with empty data for wrong artistId (inconclusive without second account having real data)

## Temp Email Registration Pattern
```python
# mail.tm API for disposable emails
POST https://api.mail.tm/accounts  {"address": "x@domain", "password": "y"}
POST https://api.mail.tm/token     {"address": "x@domain", "password": "y"}
GET  https://api.mail.tm/messages  (Authorization: Bearer <token>)
GET  https://api.mail.tm/messages/{id}  (full message with code)
```

Code format: 6 alphanumeric chars (e.g., `9H84MZ`, `U4CC4Y`)
Code expiry: 24 hours (per email body)
