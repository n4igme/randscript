## Pitfalls

### Mobile-Specific
- **Meta/Instagram apps reject login on repackaged APKs** — server-side signature validation checks the APK signing cert hash in API requests. Patching NSC + Frida Gadget + signature spoof via PackageManager hook does NOT work. The only viable approach for authenticated testing: install original APK → login → attach Frida to running process (no respawn). Accept partial traffic interception or skip to runtime testing (deep links, exported components).
- **`allowBackup=true` is Info, not Medium** — Android 12+ encrypts backups by default. Only escalate if sensitive data in plaintext SharedPreferences AND targetSdk < 31.
- **WebView `loadUrl()` ≠ XSS** — unless `setJavaScriptEnabled(true)` AND user-controlled URL/content reaches it. Check both conditions before reporting.
- **Root/jailbreak detection bypass is NOT a finding** — it's a means to test. Phase 3 is a tool, not a result. Only report if bypass enables a real attack chain.
- **Frida 16.x CLI flags changed:** `--no-kill` and `--no-pause` are removed in Frida 16+. Use the Python API (`device.spawn()`, `session.attach()`, `device.resume()`) instead of CLI for persistent sessions. The CLI in 16.x hangs without PTY.
- **Frida Python background sessions need file-based logging:** Background processes with `threading.Event().wait()` may not flush stdout to the terminal tool. Write status to a log file (`/tmp/frida_status.log`) alongside print for reliable monitoring via `cat`.
- **Production builds require Magisk su:** `adb root` fails with "cannot run as root in production builds". All privileged commands need `adb shell "su -c 'command'"` wrapper via Magisk.
- **redsocks + Burp = CONNECT-by-IP failure for Flutter:** redsocks sends `CONNECT <IP>:443` (not hostname). Burp can't generate per-host certs or log traffic. 27 ESTABLISHED connections + 0 HTTP history = this bug. See phase4-traffic.md for solutions.
- **Don't over-engineer Frida when you have source code + keys** — if you've already extracted RSA keys and mapped the auth flow from source, write a standalone Python script (requests + cryptography) instead of spending hours hooking obfuscated native SDKs via Frida. Frida is for extraction/bypass; once you HAVE the secrets, use them directly. The user's device is for capturing ephemeral values (eversafe tokens, OTPs) — not for running the entire auth flow through instrumentation.
- **Flutter plugin classes not in main classloader** — Flutter apps load native plugin code (Kotlin/Java) lazily via MethodChannel. Classes like `com.dkatalis.cipher.CipherImplementation` won't be found with `Java.use()` at attach time. Solution: use **spawn mode** (`-f`) and hook at JCE level (`javax.crypto.Cipher.doFinal`, `java.security.Signature.sign`) instead of app-specific classes. These JCE hooks catch the plugin's operations when the Flutter engine initializes. Alternatively, hook `FlutterJNI.invokePlatformMessageResponseCallback` to intercept all Dart↔Native replies.
- **Frida on KernelSU 0.7.1 (kernel 4.4)** — too old for Zygisk/Shamiko. Use hluda-server at `/data/local/tmp/`, not regular frida-server. Spawn mode (`-f`) more reliable than attach.
- **Frida `--pause` flag on Meta apps** — causes process termination. Always use spawn without `--pause`.
- **Native BoringSSL hooks on Meta apps crash the process** — hooking `dlopen`/`android_dlopen_ext` or enumerating Conscrypt exports causes SIGBUS during native lib loading. Stick to Java-layer TrustManagerImpl hooks combined with patched APK.
- **Certificate pinning bypass alone is Medium at best** — only escalates if you can demonstrate real API abuse (IDOR, auth bypass) through the unpinned traffic.
- **Flutter 3.22+ (compileSdk 36, 2026 builds) breaks ALL known SSL bypass patterns** — both the mtest `flutter_ssl_bypass.js` (pattern 0: `FF 03 05 D1 FD 7B 0F A9`) and NVISO's `disable-flutter-tls.js` (May 2025) fail to locate `ssl_verify_peer_cert`. The function prologue changed. Pattern-matched functions are never called during TLS handshake. Requires Ghidra RE + Frida backtrace approach to find the actual function via control flow analysis. See `references/flutter-ssl-bypass.md` → "Fallback: Ghidra RE + Frida Backtrace". The BoringSSL error string `CERTIFICATE_VERIFY_FAILED` is referenced via error code table (not direct pointer/ADRP), so string xref tracing doesn't work either.
- **redsocks port mismatch = "bypass works but no traffic"** — redsocks config may point to wrong Burp port (e.g., 8888 when Burp listens on 8081). Always verify with `lsof -i -P | grep java | grep LISTEN` on host. Symptom: Frida SSL bypass loads successfully, app runs, but Burp history stays empty. Fix: update redsocks.conf `port` field and restart redsocks.
- **Eversafe verification token forgeable (TLV structure, no crypto)** — Jago/banking apps using Eversafe (kr.co.everspin) enforce `x-eversafe-verification-token` header on most auth endpoints. Empty/missing/garbage values return 401. BUT the server only validates the TLV structure — a manually constructed token with correct tags (version byte 0x02, status=0x01, matching device_id, fresh timestamp) passes verification. **Forge with Python** (see `references/eversafe-attestation.md` → Token Forgery). Key rules: device_id in token must match `x-device-id` header, token TTL ~30s (regenerate each request), login endpoint skips eversafe entirely. This means full API testing is possible without the real SDK or a physical device. **HTTP/2 required** — stg-mobile.jago.com returns Cloudflare 400 on HTTP/1.1; use Burp `send_http2_request` or Python httpx.
- **Eversafe detects Frida via thread name scanning + external /proc/net/tcp port scan** — Eversafe (kr.co.everspin) has two detection phases: (1) Thread name scan `/proc/<pid>/task/*/comm` at ~10s — bypassed by renaming gmain→Binder:main, gdbus→Binder:dbus, gum-js-loop→Chrome_IOThread, pool-ggbond→pool-3-thread-1. (2) External `/proc/<pid>/net/tcp` scan from EversafeService (separate process) at ~50-60s — finds port 27042 (0x69A2, hluda listening socket), sends SIGKILL externally. hluda hides /proc/maps but can't hide its listening port from external readers without breaking USB transport. **Result:** thread rename extends survival from 10s → 53s. Full persistent bypass requires kernel-level hiding (Zygisk+Shamiko, not available on kernel 4.4). 53s window is usable for one-shot operations. For persistent interception, use HTTP Toolkit (VPN-based, no Frida needed). See `references/eversafe-frida-bypass.md` for architecture details, scripts, and what doesn't work.
- **hluda-server with non-default port breaks Frida USB transport** — starting `hluda-server -l 127.0.0.1:31337` to hide port 27042 causes `get_usb_device().spawn()` to fail with "need Gadget". Workaround: use `get_remote_device('127.0.0.1:27042')` with `adb forward tcp:27042 tcp:31337`. This WORKS for spawn/attach but is actually MORE detectable by Eversafe (10s survival vs 53s with USB). The adb forward creates additional TCP socket pairs visible in /proc/net/tcp. **Conclusion:** custom port + adb forward is WORSE than default port + USB transport. Don't bother.
- **HTTP Toolkit as Frida-free interception for anti-tamper apps** — when Eversafe/AppFence kills Frida-attached processes, HTTP Toolkit (VPN-based) provides persistent traffic interception without triggering anti-tampering. It handles SSL bypass at the VPN layer, not by instrumenting the app process. Use HTTP Toolkit for persistent interception + Frida (53s window) for on-demand hooks. HTTP Toolkit works where Burp+redsocks+Frida fails because: (1) no Frida process attachment = no detection, (2) VPN handles HTTP/2 properly (redsocks CONNECT tunnel has compatibility issues with some Flutter HTTP/2 implementations), (3) no iptables/redsocks port mismatch issues.
- **Binary patching libflutter.so crashes Flutter engine** — replacing `ssl_verify_peer_cert` with `MOV W0, #0; RET` in the .so file causes SIGSEGV during `FlutterJNI.performNativeAttach` (fault addr 0x6a8, null pointer). The Flutter engine or Eversafe validates .so integrity at load time. Patching the binary is NOT viable when integrity checks are present. Stick to Frida runtime hooks.
- **App UID changes after uninstall/reinstall** — iptables rules use UID. After `adb uninstall` + `adb install`, the UID increments (e.g., 10464→10465→10466). Always re-check with `pm list packages -U | grep <pkg>` and update iptables rule.
- **macOS Burp + iptables DNAT = no traffic** — Burp on macOS cannot read `SO_ORIGINAL_DST` from DNAT'd connections. Solution: use redsocks on the Android device (reads `SO_ORIGINAL_DST` locally, sends CONNECT to Burp). See `references/redsocks-transparent-proxy.md`.
- **redsocks + Burp listener settings** — redsocks `type = http-connect` sends plain HTTP CONNECT requests. Burp listener MUST have: Force TLS OFF, all interfaces ON. If Force TLS is on, Burp sends TLS alert (handshake_failure).
- **System cert on system-as-root devices doesn't persist across reboot** — writing to `/system/etc/security/cacerts/` via `mount -o rw,remount /` works until reboot. Must re-install after every boot. Consider using Magisk/KernelSU module for persistent cert injection instead.
- **GhidraMCP duplicate extension collision** — if GhidraMCP is installed in BOTH `/opt/homebrew/Cellar/ghidra/.../Extensions/` AND `~/Library/ghidra/.../Extensions/`, Ghidra crashes with "Multiple modules collided with same name: GhidraMCP". Fix: remove one copy. Also: `Module.manifest` must follow valid format (`MODULE DIR: lib` on first line), not arbitrary key-value pairs like `GHIDRA_MODULE_NAME=...`.
- **GhidraMCP REST API (port 8080) requires CodeBrowser open** — the GhidraMCP extension only starts its HTTP server when a binary is opened in the CodeBrowser window. Just launching Ghidra GUI is not enough. The `mcp_ghidra_*` tools won't work until you open a project AND double-click a binary to analyze. Headless mode (`analyzeHeadless`) does NOT start the REST server — use headless scripts for automated analysis, GUI+MCP for interactive exploration.
- **Ghidra headless + GhidraMCP bridge process** — the bridge (`bridge_mcp_ghidra.py`) connects to Ghidra's REST API on port 8080. If Ghidra GUI is killed/restarted, the bridge stays running but returns errors. After restarting Ghidra GUI and opening a project, the bridge reconnects automatically. If bridge itself dies, restart with: `python3 /opt/GhidraMCP/bridge_mcp_ghidra.py --ghidra-server http://127.0.0.1:8080/`
- **System CA install doesn't help Flutter** — Flutter's BoringSSL in libflutter.so does NOT use Android's /system/etc/security/cacerts/ store. Installing Burp CA as system cert only fixes Java-layer SDKs (CleverTap, Datadog, OkHttp). Flutter Dart HttpClient still rejects Burp's cert. You need a working Frida hook on the actual verify function + redsocks.
- **Always verify Flutter SSL bypass hooks are triggered** — after installing hooks, add `console.log` in `onEnter`. If never called during app network activity, the pattern matched wrong functions. Don't assume "SSL bypass active (N hooks)" means it's working. Fall back to Ghidra RE approach in `references/flutter-ssl-bypass.md`.
- **QUIC (HTTP/3) bypasses HTTP proxies entirely** — apps using QUIC (UDP 443) won't route through Burp/mitmproxy. Block with `iptables -A OUTPUT -p udp --dport 443 -j DROP` to force TCP/TLS fallback. Meta, Google, and Cloudflare apps commonly use QUIC.
- **`exported=true` on activities ≠ vulnerability** — only if the activity handles sensitive data without re-authentication or accepts untrusted input that leads to harm.
- **`overridePins="true"` + user CA in NSC base-config = Phase 3 pass-through** — if `network_security_config.xml` has `<certificates overridePins="true" src="user" />` in `<base-config>`, SSL pinning bypass via Frida is unnecessary. Just install proxy CA as user cert. Check NSC in Phase 2 FIRST — saves hours of bypass work. Instagram v431+ has this despite 18 SHA-256 pins in pin-set. The `overridePins` attribute nullifies all pins for user-installed CAs.
- **Burp "Support invisible proxying" REQUIRED for iptables DNAT** — when using DNAT to redirect Flutter/app traffic to Burp, the listener MUST have: (1) bind to "All interfaces" (not loopback), (2) "Support invisible proxying" enabled under Request handling. Without invisible proxy, DNAT'd connections establish (visible in `lsof`) but produce zero HTTP history entries because Burp expects a CONNECT request from explicit proxy clients. Invisible mode auto-detects TLS via SNI and generates per-host certs. Diagnostic: `iptables -t nat -L OUTPUT -n -v` shows packet counters incrementing + `lsof -i :8080 | grep ESTABLISHED` shows device connections = DNAT works but Burp config wrong.
- **Device lock screen blocks dynamic testing** — verify device is unlocked before deep link / activity launch testing. `adb shell dumpsys window | grep mShowingLockscreen` to check. Set `adb shell settings put system screen_off_timeout 600000` early. Screenshot size is the oracle: 14KB = screen off/locked, 1-2MB = real content. If PIN-locked and PIN unknown, flag immediately — don't waste cycles on repeated launches.
- **Meta apps: deep link scheme is `instagram://` not `ig://`** — despite manifest declaring both `ig` and `instagram` schemes, runtime resolution works with `instagram://` for most hosts (e.g., `instagram://fbpay_hub`). `ig://` scheme fails to resolve at runtime. The 426 deep link hosts use the UrlHandlerActivity with `https://instagram.com/*` for web links.
- **Threads (com.instagram.barcelona) uses `barcelona://` scheme** — NOT `instagram://`. Does NOT have `extbrowser` host. Has unique hosts like `inject`, `inter_app_redirect`, `slide_*`, `group_invite_link_deeplink`. Requires login before deep links route properly (all redirect to login screen without auth). See `references/meta-instagram-testing.md` → "Threads Specifics".
- **Meta exported FlipperDiagnosticActivity** — `com.facebook.flipper.android.diagnostics.FlipperDiagnosticActivity` is exported in production Instagram builds. Launches from any app via `am start`. Requires app in foreground + Flipper desktop connected to show content. Document as attack surface (debug tool in prod) but verify actual data exposure before reporting.
- **Threads (com.instagram.barcelona) deep link content injection** — `barcelona://create?text=<URL-encoded text>` pre-fills the post composer with attacker-controlled text including phishing URLs and @mentions. The scheme is BROWSABLE (triggerable from web pages via `<a href="barcelona://create?text=...">` or JS redirect). The app renders link previews for attacker URLs. User must still tap "Post" (no auto-post), so severity is Low-Medium. PoC: HTML page with link to `barcelona://create?text=%40zuck%20free%20crypto%20https%3A%2F%2Fevil.com`. Same vector likely exists on Instagram via `instagram://create` — test separately.
- **Threads (barcelona) Frida attach fails same as Instagram** — hluda-server 16.1.8 cannot attach or spawn com.instagram.barcelona. Both attach (`-p PID`) and spawn (`-f package`) fail with "connection closed". Same anti-Frida as Instagram. No workaround found for kernel 4.4 / KernelSU 0.7.1 device.
- **Threads deep links: `slide_*` hosts trigger account switch prompts** — `barcelona://slide_thread`, `barcelona://slide_create_chat` etc. show "Log in / Switch profiles" dialog instead of navigating. They appear tied to a different profile context. Not directly exploitable but causes UI disruption.
- **Threads `chat.threads.com` URLs fall through to Chrome** — domain doesn't resolve publicly. Threads registers it in manifest but can't handle it in-app, so it opens Chrome with the full URL. Not an open redirect (Chrome loads the original URL, not a parameter). Same for unrecognized paths on registered domains.
- **Threads NSC identical to Instagram** — `overridePins="true" src="user"` in base-config, 18 SHA-256 pins in domain-config for Meta domains. SSL pinning bypassable with user-installed CA cert. `cleartextTrafficPermitted="true"` in base-config.
- **Content providers with signature protection despite exported=true** — Meta apps mark providers as `exported="true"` but enforce signature-level checks at runtime (`SecurityException: Component access not allowed`). Don't report exported providers as findings without confirming actual data access from a third-party caller.
- **`exported=true` + `enabled=false` = not testable without auth** — Meta apps export components but disable them at manifest level. They're enabled at runtime only after login. Check `android:enabled` before spending time on exported components.
- **BROWSABLE deep link content injection (create/compose flows)** — Social media apps that register BROWSABLE deep links for post creation (e.g., `barcelona://create?text=`) allow web pages to pre-fill the composer with attacker text. Test pattern: (1) identify create/compose/share deep link hosts in manifest, (2) test with `?text=` param via adb, (3) if text appears in composer, validate full chain with `scripts/deeplink_browsable_chain_test.sh`. This is a class of finding across social apps — not just Meta. Instagram's `create_post` opens gallery (no text injection) but Threads' `create` accepts text directly.
- **Instagram `create_post` / `share` deep links open gallery picker only** — Instagram posts require media. These deep links open the photo/video selector, NOT a text composer. No text pre-fill injection possible on Instagram via deep links. Threads is different because it's a text-first platform.
- **Server-side APK signature validation (Meta/banking apps)** — Some apps validate the signing cert server-side during login. Repackaged APKs cannot authenticate regardless of client-side PackageManager hooks. Workaround: login on original APK, preserve app data dir, swap to patched APK. See `references/meta-instagram-bypass.md`.
- **Don't report debug/test code in release builds as High** — unless it's actually reachable (check if BuildConfig.DEBUG gates it). Unreachable dead code is Info.

### Response Length
- **Keep responses SHORT and split across turns** — long tool outputs, base64 blobs, and multi-step summaries cause "Response truncated due to output length limit" errors. Never dump full keys, full API responses, or multi-hundred-line outputs in a single response. Summarize, then offer details on request.
- **execute_code scripts must print selectively** — filter output to relevant lines only. Never `print(output)` of full curl/frida output. Extract and print only the signal (status codes, tokens, key findings).

### Workflow
- **NEVER sync FROM randscript/myherms TO Hermes skills without explicit user confirmation.** Hermes is source of truth.
- **Frida CLI `-q` flag exits immediately after script load** — hooks are injected but Frida detaches within seconds, removing Interceptor hooks from the process. For persistent bypass sessions (SSL + root), MUST use Frida Python bindings with a `time.sleep()` loop or `sys.stdin.read()` to keep the session alive. The `-q` flag is only useful for fire-and-forget scripts that patch memory (not Interceptor hooks).
- **Frida CLI with background/tee has buffering issues** — `console.log()` output doesn't flush to file reliably when using `tee` or `-o`. For coordinating Frida hooks + adb deep link triggers, use Frida Python bindings via `execute_code` with `subprocess`:
  ```python
  import frida, subprocess, time
  device = frida.get_usb_device()
  session = device.attach(pid)
  script = session.create_script(js_code)
  script.on('message', on_message)
  script.load()
  subprocess.run(['adb', '-s', serial, 'shell', 'am', 'start', ...])
  ```
  This gives reliable message capture via `send()` + `on_message` callback.
  NOTE: `frida` Python module is NOT in `execute_code` sandbox — use `terminal()` with `python3 -c "..."` or a standalone script file.
- **Frida CLI `-q` (quiet) exits after script load** — no REPL keeps session alive, hooks are lost on detach. For persistent bypass during dynamic testing, write a Python launcher with `while True: time.sleep(1)` keep-alive and run as background process. Frida 16.1.8 also does NOT support `--no-pause` flag; `-f` spawn auto-resumes.
- **Content providers with `exported=true` + `enabled=false`** — even after login enables them, Meta apps enforce runtime caller signature validation ("Component access not allowed... Called by: com.android.shell"). Only Meta family apps (Facebook, Threads, WhatsApp) pass. Not exploitable from third-party apps.
- **Scripts from delegate_task subagents are not recoverable from state.db** — rebuild from scratch if lost.
- **NEVER slice output strings in execute_code** (e.g. `print(r["output"][:500])`) — user requires full untruncated output. Use separate `terminal()` calls per command, each printing a manageable portion. One-by-one sequential calls are always safer than batching with sliced prints.
- **Before bulk file operations, verify backup exists.**
