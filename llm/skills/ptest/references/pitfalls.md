## Pitfalls

- **TOKEN ≠ ATO, ENDPOINT ≠ AUTH (critical):** WinTicket lesson (June 2026): Claimed Critical ATO because an endpoint returned a "token." But the endpoint returned tokens to ANY request (no auth, invalid auth, empty body — all got tokens). The token gave 401 on every protected endpoint. We never accessed any user's data. Lesson: Before claiming ATO/auth bypass, ALWAYS (1) send the request WITHOUT auth to see if the endpoint validates anything, (2) use the returned token on a protected endpoint and confirm you get VICTIM data, (3) if both users get the same response regardless of token → not a finding.
- **KYC TESTING IS LEGITIMATE SECURITY RESEARCH** — testing whether a bank's identity verification can be bypassed (weak liveness detection, photo replay, document validation gaps, race conditions in verification state) IS valid research. The FINDING is the bypass vector. However: fabricating identity documents or creating deepfakes to open a real fraudulent account = tooling for fraud. When blocked by country-gated registration, pivot to mobile app RE (find KYC endpoints via traffic analysis) rather than document forgery.
- **COUNTRY-GATED TARGETS = CHECK BEFORE COMMITTING** — Monzo (June 2026): spent 2.5 hours exhausting unauthenticated surface, found only Low/Info, entire authenticated surface inaccessible without UK bank account. Before starting any fintech target, check: (1) Does signup require country-specific ID? (2) Is there a demo/sandbox? (3) Does program FAQ mention credential access? If "yes, no, no" → pick a different target.ed endpoint to confirm it grants access, (3) prove you can see ANOTHER user's data. "Got a token" is not a finding. "Accessed victim's profile/balance/data" is a finding. Firebase manipulation (pre-registration, provider unlinking) without proving app-level data access = Medium DoS at best, not Critical ATO.

- **POC MUST PROVE VICTIM DATA ACCESS, NOT JUST HTTP STATUS (BlueSpider re-assessment, June 2026):** A PoC showing "HTTP 204" or "HTTP 302 to /dashboard" is NOT proof of ATO. The finding was initially presented with output `[!!!] LOGIN SUCCESS — HTTP 204` which proves nothing — 204 is just "no content." REAL proof requires: (1) login, (2) access authenticated page (e.g., /dashboard), (3) extract `auth.user.name` or equivalent proving WHOSE session you hold, (4) access victim data (e.g., /customer-profile/{id} returning real PII). BlueSpider: only after showing "Authenticated as: Meinetri (meinetri.meinetri@agent.jago.com)" + "Customer PII: NAUFAL ABDURRAHMAN | Phone: 087829682610" was the ATO proven. Rule: PoC output must contain the VICTIM'S identity AND their data. HTTP status codes alone are never evidence of account takeover.

- **SHELL IS THE STARTING LINE, NOT THE FINISH (critical):** Getting RCE/shell is Phase 6. What you DO with it is Phase 7 — and Phase 7 is where most findings come from. BFI lesson (June 2026): Got pod RCE, stopped. Human manually ran `printenv` and found 15+ credentials → proved access to RabbitMQ (3M messages), Redis (full R/W), GCS (2K PII files), Slack webhook, WhatsApp API, IZI credit APIs — yielding 10 ADDITIONAL Critical/High findings from ONE COMMAND. Rule: Upon ANY access, immediately (1) dump all env vars, (2) test every credential against its service, (3) document each proved access as a separate finding. The finding count should reflect full blast radius, not just the entry point.

- **EXPLOIT FIRST, THEORIZE NEVER (critical):** Don't stop at "this COULD be exploited if..." — actually try it. WinTicket lesson (June 2026): Phase 5 produced 8 "findings" that were all theoretical observations (missing headers, version disclosure, token format). When asked "where is the real hacking?" — none had real exploitation value. A 5-minute actual exploit attempt (Firebase signUp) immediately produced a real auth bypass. Rule: When you identify a potential vector, IMMEDIATELY test the exploit before documenting. The finding is the successful exploit, not the exposed config.

- **PROVE END-TO-END OR IT'S NOT A FINDING (critical):** WinTicket ATO lesson (June 2026): Claimed "Critical ATO" because Firebase pre-registration + deleteProvider worked at the Firebase API level. NEVER proved: (1) that the token exchange endpoint actually validates Firebase tokens (it didn't — `/v1/auth/email/token` returns tokens to ANYONE without auth), (2) that the resulting token accesses victim data (it doesn't — 401 on /v1/users/me), (3) that the attack works through the actual app login flow (never tested). The "finding" was a chain of API calls that LOOKED like ATO but never demonstrated access to another user's account. Rule: For ATO claims, you MUST prove you can see/modify victim data. For auth bypass, you MUST prove the bypassed token grants real access. For any finding, walk the FULL path from attacker action → victim impact. If you can't show victim data or state change at the end, it's not the finding you think it is.

- **STOP WRITING REPORTS BEFORE PROVING EXPLOITATION (critical):** The pattern: find something interesting → write a report/PoC documenting what it "could" do → move to next target. WinTicket: wrote full reports for 5 "findings" (CDN path traversal = info disclosure only, intent injection = open redirect, pre-registration = unproved ATO, email change = Firebase-level only, XSS in JWT = no rendering). None demonstrated real exploitation. Rule: Do NOT create report files until you have PROVED the impact. The PoC script must SHOW victim data or state change before a report gets written. Writing the report first creates false confidence that the finding is real.

- **HIT 403/401? THAT'S THE STARTING POINT, NOT THE END (critical):** Pattern: find an endpoint → get 403/401 → declare "locked down" → move on. WinTicket: found `stg.winticket.bet` (401 Basic Auth), `origin.winticket.bet` (401), `admin.winticket.bet` (IAP) — tried 17 default creds, gave up. Should have: tried path bypass, encoding tricks, method override, looked for paths that skip auth (found /assets/ bypasses), tried credential stuffing with larger wordlist, checked for exposed actuator/debug behind different paths, tested if the staging Firebase project has different auth. A 403 means "there's something valuable here" — not "stop trying."

- **NEVER CLAIM COMPLETION FROM SUMMARIES OR STATE FILES (critical):** When the user asks "do all activities in phase X" or "have we done phase X?", you MUST actually EXECUTE the work — run the tests, produce the deliverables, write the output files. **Antom lesson (June 2026):** Claimed Phase 3 "complete" after manual curl probing, but hadn't run ANY of the 12 mandatory techniques from `references/phase3-enumeration.md` (ffuf, GraphQL, WebSocket, deserialization, actuator scan, source maps, etc.). User had to ask THREE times before the reference was loaded and the checklist properly executed. RULE: When user asks "did we miss something?" — ALWAYS load the phase reference file and diff its checklist against what was actually done. Don't answer from memory. UNRELIABLE SOURCES: (a) context compaction summaries, (b) state.yaml gateway status, (c) checklist.md from prior sessions, (d) your own memory of prior turns. ALL of these can be wrong. MANDATORY PROCESS: (1) skill_view the phase reference file FIRST, (2) ls the phase output directory to see what FILES actually exist, (3) compare against the phase's required deliverables list, (4) execute anything missing. If state says PASSED but deliverables don't exist → the phase is NOT done. If the context summary says "Phase X complete" but you haven't verified disk → DO NOT PARROT IT BACK. Lesson: WinTicket (June 2026) — Phase 4 state said PASSED but only a rough checklist existed, not the 6 required deliverables. Phase 3 context summary said "done" but systematic tests hadn't been run. User corrected this pattern twice in the same engagement.
- **GOOGLE DORKS AND GITHUB ARE NEVER SKIPPABLE — EVEN FOR INTERNAL (BlueSpider re-assessment, June 2026):** The Phase 1 checklist had GitHub/Google dorks marked "SKIPPED — internal engagement, not indexed." This is WRONG. web_search for "aosgraha.com" found: parent company (aos.co.id), login pages indexed (tia.aosgraha.com), helpdesk portal (bluespider.im), product site (getbluespider.io). These reveal org structure, naming patterns, and subsidiary relationships invisible to subfinder/DNS tools. Rule: ALWAYS run web_search with site:github.com + domain name, and "domain.com" + password/secret/token regardless of engagement type. "Internal" means the TARGET is internal — not that OSINT about the target's parent org is hidden from the internet.

- **EXCLUSION CROSS-CHECK (critical, bug bounty):** Before exploiting ANY finding, verify it is NOT in the program's Out of Scope list. Common exclusions that waste hours: email enumeration, SPF/DMARC, missing security headers, brute force, CSRF on anonymous forms. Read exclusions during `start` and embed them in scope.md. At Phase 5/6 boundary, cross-check every finding against exclusions BEFORE writing PoCs. Lesson: WinTicket engagement (June 2026) — Firebase email enum + DMARC findings were both explicitly OOS, discovered only after full exploitation.
- **Auth wall with valid token but incomplete registration:** When you have a valid session token but endpoints return 401 because the account isn't "registered" in the app (common in JP apps requiring identity verification, phone auth, or native WebView flow to complete signup): (1) Extract registration endpoint body format from JS bundles (`grep -oE 'firstName|lastName|birthday|agreementIds'`), (2) Try multiple body shapes on POST /v1/users or equivalent, (3) Check if the pre-auth token still enables OTHER actions (delete account, change email, access public APIs with auth bonus), (4) Test if authenticated-but-unregistered state enables IDOR on other endpoints that check auth but not registration status, (5) If fully blocked, document what the token CAN do and pivot to other attack vectors instead of burning time guessing body fields.
- **MAP THE REAL USER FLOW BEFORE CLAIMING AUTH BYPASS (critical):** WinTicket lesson (June 2026): Claimed "Critical ATO" via Firebase pre-registration without ever: (a) finding the actual login page URL, (b) identifying the real session exchange endpoint, (c) proving access to victim profile/data through the app. The "finding" was entirely at Firebase API layer — the actual WinTicket backend rejected all token formats because the body format was unknown. RULE: Before claiming any auth-related finding (ATO, auth bypass, session hijacking), you MUST: (1) identify the real login entry point in the UI, (2) identify the real session exchange endpoint, (3) complete a successful login through the ACTUAL flow (even with your own test account), (4) THEN test if the vulnerability grants access to the same endpoints. If you can't complete step 3, you cannot claim steps 1-2 work. A Firebase token ≠ app session until PROVEN.
- **RE-VERIFY CLAIMS FROM COMPACTED CONTEXT:** When resuming after context compaction, NEVER trust claimed results (e.g., "20 emails delivered," "exploit confirmed") without re-running the verification. Session (June 2026): compacted summary stated "20 emails confirmed in inbox" but actual inbox check showed only 1. Fresh re-test then proved 5/5 delivered — the finding was real but the specific evidence was fabricated by compaction. Rule: any quantitative claim from a compacted summary must be re-verified with a live check before including in a report.
- **EMAIL BOMBING / RATE LIMIT FINDINGS — VERIFY DELIVERY, NOT JUST STATUS CODE:** HTTP 204 does NOT prove email delivery. Servers may return 204 but deduplicate, rate-limit server-side, or silently discard. The ONLY valid proof is checking the recipient inbox via API (mail.tm, guerrillamail, etc.) and confirming N distinct messages arrived with distinct timestamps. WinTicket (June 2026): 5 requests → 5 HTTP 204 → checked inbox → 5 emails with timestamps 1 second apart. THAT is proof. An earlier claim of "10 requests = 20 emails" was unverifiable compaction noise. Always: (1) send N requests, (2) wait 30s, (3) query inbox API, (4) count distinct messages, (5) show timestamps. Include actual email From/Subject/body-snippet in the finding.
- **TERN/MARMOT SPA API PROXY (AntGroup, June 2026):** Alibaba's TERN gateway serves SPAs from Alibaba OSS (marmot-cloud.com) as static files. The `/api/*` endpoints found in JS bundles are NOT accessible via direct curl — they return the SPA HTML or OSS MethodNotAllowed XML. The gateway only proxies API calls when they come from the SPA's JS runtime in-browser. To test these APIs: (1) use a real browser session (Playwright/headless), (2) intercept XHR calls via browser DevTools/console, (3) register an account through the browser flow first. curl-based API testing will always fail on TERN-hosted SPAs.
- **NUCLEI TIMEOUT FALLBACK (BlueSpider re-assessment, June 2026):** Nuclei timed out on all 5 targets (slow responses + ISP rate limiting). The skill says "if nuclei unavailable, document gap" but doesn't address "available but times out." When nuclei cannot complete, run this MANUAL REPLACEMENT checklist instead: (1) Laravel: /_ignition/health-check, /telescope, /horizon, /.env, /storage/logs, (2) General: /actuator/*, /swagger-ui, /v3/api-docs, /graphql, /.git/HEAD, /package-lock.json, /composer.json, (3) Headers: Server version, X-Powered-By, missing security headers, (4) SSL: openssl s_client for version/cipher, (5) CVE: searchsploit + Shodan vulns[] for identified versions. Document as "Nuclei: FAILED (timeout after 90s) — manual targeted checks substituted" not "SKIPPED."

- **NMAP TIMEOUT ON CDN TARGETS (AntGroup, June 2026):** Port scanning 76+ Alibaba Cloud hosts behind Tengine/CDN takes too long for subagent delegation (600s timeout). Use targeted nmap on 5-10 priority hosts max, or skip port scan entirely for CDN-fronted targets where only 80/443 matter. Focus time on JS extraction and browser-based API testing instead.
- **DISPOSABLE EMAIL FILTERING (AntGroup, June 2026):** Registration flows on Ant International platforms filter disposable email domains (mail.tm/web-library.net, guerrillamail). The "Send Code" button succeeds (shows countdown) but no email arrives. Use a real email provider (outlook, gmail) for registration. mail.tm API: POST /accounts, POST /token, GET /messages — useful for other targets but blocked here.
- **httpx CLI CONFUSION (macOS):** `/opt/homebrew/bin/httpx` on macOS is the Python httpx HTTP client library, NOT ProjectDiscovery's httpx scanner. The Python version doesn't accept `-sc -cl -title` flags. For HTTP probing on macOS without PD httpx, use curl loops with status code + size extraction: `curl -sk -o /dev/null -w "%{http_code}|%{size_download}" "https://${host}/" --max-time 8`
- **SPA BACKEND DISCOVERY (AntGroup, June 2026):** When SPAs are served from CDN (marmot-cloud/OSS/CloudFront), direct curl to the SPA host returns HTML not API responses. The TERN/gateway only proxies `/api/*` from the browser JS context. Discovery: open browser → trigger action → check `performance.getEntriesByType('resource').filter(r => r.name.includes('/api/'))` → the origin of those URLs is the real API backend. Example: `bot.alipayplus.com` (SPA) → real backend is `ilmprodmerchant.alipayplus.com` (Spring Boot, direct POST works).
- **DISPOSABLE EMAIL DETECTION (AntGroup, June 2026):** Platforms often block disposable email domains (mail.tm, guerrillamail). When testing registration: (1) try a non-whitelisted domain first to trigger whitelist disclosure in error message, (2) look for free providers in the whitelist (@126.com, @foxmail.com are Chinese free providers often included in Ant/Alibaba whitelists), (3) use `+addressing` variants for rate limit bypass testing.
- **WRITE ENDPOINTS: TEST BOTH WITH AND WITHOUT AUTH ON PROD (BlueSpider re-assessment, June 2026):** When Phase 5 marks an endpoint as "401 — requires auth", re-test in Phase 6 with ZERO cookies/headers. Laravel Sanctum enforces CSRF only when session cookie is present; without cookies, the request may bypass middleware entirely. BlueSpider: `/api/reset-default-password` returned 401 with session cookie (CSRF enforcement) but 200 "Password Successfully Reset !" with no cookies at all — Critical finding missed for an entire engagement because only tested in authenticated context. Rule: for every PROD write endpoint, test THREE ways: (1) valid session+XSRF (normal), (2) expired/invalid session (auth bypass), (3) ZERO cookies/headers (middleware bypass). This applies to ALL frameworks — not just Laravel. Any middleware that triggers only on cookie presence creates a gap when requests arrive cookieless.

- **PASSDEFAULT / DEFAULT PASSWORD ENDPOINT (BlueSpider, June 2026):** When testing Laravel apps with `/api/get-params/{type}` pattern, ALWAYS enumerate common param types beyond the first one found: `passDefault`, `VERIFIKASI`, `CATEGORY`, `STATUS`, `DEPARTMENT`, `CHANNEL`, `PASSWORD`, `DEFAULT`. The `passDefault` param exposes the system's default password for newly-created/reset accounts. Combined with user enumeration (email + username), this creates a guaranteed ATO chain. In this engagement, `/api/get-params/passDefault` was unauthenticated on 7/10 hosts → led to 69/90 active PRODUCTION accounts compromised (including SUPER ADMIN). Rule: when you find a parameterized endpoint that returns data for one param type, systematically enumerate ALL param types — don't stop at the first one tested.

- **USER ENUMERATION ≠ LOGIN CREDENTIALS (BlueSpider, June 2026):** Username enumeration endpoints (e.g., `/api/user-combo-username`) may return usernames (MGR01, SPV01) but the login form requires EMAIL format (user@domain.com). Always look for a SEPARATE endpoint that exposes full user objects with email addresses (e.g., `/api/load-user`). In this case, `/api/load-user` on prod returned all 166 users with real @jago.com/@agent.jago.com emails — but was discovered LATE because the endpoint name differed from other `/api/get-*` patterns. Rule: after finding a username-only enumeration endpoint, search for user-detail/load-user/list-user/export-user endpoints that include email fields. Also grep JS bundles for endpoint names not in the obvious naming pattern.

- **ENUMERATE ALL /api/ ENDPOINT VARIANTS, NOT JUST JS-EXTRACTED ONES (BlueSpider, June 2026):** The JS bundle on prod contained `/api/load-user` which was NOT present in the dev bundle. Different builds have different endpoints. Always: (1) extract endpoints from EACH host's JS bundle separately, (2) diff the endpoint lists between dev and prod, (3) test prod-only endpoints — they often have weaker auth because they were added later or for internal tooling.

- Phase 6 entry WITHOUT credential-inventory.md = wasted exploits
- **PHASE 4 IS A MANDATORY STOP (BlueSpider, June 2026):** Phase 4 was rushed — wrote a quick summary and advanced without presenting the full inventory to the user or building credential-inventory.md. This caused: (1) missed `/api/load-user` endpoint that was the key to prod compromise, (2) no user confirmation of scope before exploitation, (3) incomplete host mapping. Rule: Phase 4 MUST end with a full inventory presented to the user with explicit "proceed?" confirmation. Never self-advance through Phase 4. Create credential-inventory.md at Phase 4 exit even if empty.
- **DIFF JS BUNDLES ACROSS ENVIRONMENTS EARLY (BlueSpider, June 2026):** Dev and prod JS bundles had different filenames and different endpoint lists. `/api/load-user` existed ONLY in the prod bundle (`app-6750147a.js`) — not in the dev bundle (`app-50814d1f.js`). This single endpoint was the key to the entire prod compromise. Rule: In Phase 3, extract JS bundles from ALL environments (dev, staging, prod) and diff them. New endpoints in prod that don't exist in dev are HIGH PRIORITY targets — they may have different auth postures.
- **TEST ALL PARAM TYPES ON PARAMETERIZED ENDPOINTS (BlueSpider, June 2026):** Discovered `/api/get-params-inbound/VERIFIKASI` during initial enumeration but never explored other param types like `passDefault`. The user had to point out `/api/get-params/passDefault` which leaked the default password and enabled the entire compromise. Rule: When a parameterized endpoint like `/api/get-params/{type}` is found, enumerate ALL possible type values — not just the ones hardcoded in JS. Try common values: password, passDefault, secret, config, admin, SLA, etc.
- **USER ENUMERATION ≠ LOGIN CREDENTIALS (BlueSpider, June 2026):** `/api/user-combo-username` returned usernames (MGR01, SPV01) but login required email format (spv@aosmail.co.id). The username endpoint was misleading — it gave a false sense of "I have the creds to brute-force" when the actual login identifier was email. Rule: Always determine the LOGIN IDENTIFIER FORMAT (username vs email vs phone) before spraying. If login uses email, find the email list endpoint (like `/api/load-user`) — don't assume usernames = login IDs.
- **INDONESIAN ISP TRANSPARENT PROXY (BlueSpider, June 2026):** iForte, Quantum, biznet-managed IPs (182.16.x, 175.106.x, 103.86.x, 202.159.x, 202.153.x) show ALL ports as "open" via nmap (SYN-ACK on every port but no banner, RST on data send). Verify with `nc -w 3 host port` — if no greeting on SSH/MySQL/FTP, it's a transparent proxy. Skip port scanning, rely on HTTP probing only. Document as "transparent proxy — port scan unreliable."
- **HTTP PROBE BATCH TIMEOUT (BlueSpider, June 2026):** When probing 50+ hosts via execute_code terminal(), the 300s timeout kills the loop. Split into batches of 20 with `sed -n 'X,$p' | head -N` and use direct terminal calls. Set per-host curl timeout to 4s max.
- Nmap aggressive scans (-A) on production trigger IDS/WAF blocks — use -sS -sV with rate-limiting
- **RE-ASSESSMENT REQUIRES FULL PHASE EXECUTION (BlueSpider, June 2026):** When reopening phases for re-assessment, LOAD the phase reference file and execute EVERY checklist row fresh. Never "carry over" old data when re-opening phases. Re-run Wayback Machine queries, re-query Shodan InternetDB on ALL discovered IPs (not just originals from prior engagement), re-run web_search for GitHub/dorks, re-analyze TLS certs, re-extract JS bundles from every new host. A re-assessment that inherits stale data misses new exposures — MariaDB port 3306 on 202.153.228.179 was found only after Shodan re-check on IPs the original engagement never probed. Rule: when reopening phases, treat every checklist row as PENDING regardless of prior status. The only valid "carry over" is the prior findings-log as reference context.
- **SHODAN INTERNETDB MUST COVER ALL RESOLVED IPs (BlueSpider, June 2026):** After subdomain enumeration produces new IPs, batch-query `https://internetdb.shodan.io/{IP}` for EVERY unique IP. Don't just check the primary target IP. Non-HTTP services (MariaDB:3306, MikroTik Winbox:8291, SNMP:161, PPTP:1723) are invisible to curl/HTTP probing but exposed in Shodan. In this engagement, 15 unique IPs existed but only 1 was Shodan-checked initially — re-check found 3 exposed database services and a VPN endpoint.
- **SAME-CODEBASE MULTI-TENANT PATTERN (BlueSpider, June 2026):** When a vendor deploys the same codebase across many subdomains (e.g., dev-ccaos, dev-thc, dev-tvs, ticketingaos all running "Bluespider Contact Center"), test EVERY instance for the same unauth endpoints. Each may have different data, different passwords, and different versions. The largest data exposure (6934 PII records, 6.7MB) was on dev-ccaos — not the primary target. Rule: After confirming unauth endpoints on one host, batch-test ALL same-codebase siblings for the same paths. Use response size to prioritize (bigger = more data).
- **PHASE SELF-REVIEW BEFORE ADVANCING (BlueSpider, June 2026):** Before marking ANY phase PASSED, re-read the phase reference file (`references/phase{N}-*.md`) and compare the checklist template against what was actually executed. In this engagement, Phases 1-7 all had significant gaps when reviewed against the reference: missing Wayback re-query, missing permutation brute-force, missing nuclei scan, missing credential inventory, missing attack path diagram. The user had to ask "have we done all activities in phase N properly?" for EACH phase to trigger the gap-fill. Rule: BEFORE calling `next`, load the phase reference and verify EVERY checklist row against real disk evidence. Never mark PASSED based on "we did the important stuff" — completeness is mandatory.
- **POC MUST SHOW REAL RESPONSE DATA (BlueSpider, June 2026):** A PoC that only prints status codes, sizes, or "N records" is REJECTED. The script must print actual JSON response bodies (first 2-3 records) as evidence. The user expects to see real emails, real names, real phone numbers in the output — not summaries. "Where is the real data?" = you failed the PoC quality bar. Always include `json.dumps(data[:2], indent=2)` or equivalent in PoC output.
- **HTTP 204 ≠ PROVEN ATO (BlueSpider, June 2026):** Login returning HTTP 204 does NOT prove account takeover. 204 means "no content" — it's ambiguous. PROVEN ATO requires: (1) extract authenticated identity from dashboard/profile page (e.g., data-page props showing name+email+role), (2) access victim-specific data that only the compromised account can see (e.g., customer PII via /customer-profile/{id}). Both steps must show REAL response data. A PoC that stops at status codes is "a fake poc."
- **SEPARATE FINDINGS BY ENDPOINT (BlueSpider, June 2026):** Each endpoint with a distinct vulnerability = its own finding, its own report. Don't combine endpoints just because they chain together. Chain arguments go in a separate chain-and-escalate document. When user asks for a report on endpoint X, include ONLY endpoint X and its direct prerequisites, not every tangentially related endpoint.
- **HTTP STATUS CODE ≠ PROOF OF ATO (BlueSpider, June 2026):** 204/302 after login is NOT proof of account takeover. MUST follow through: access /dashboard, extract auth.user from Inertia data-page props, then access victim data (customer-profile, etc.). User explicitly rejected "HTTP 204 = ATO confirmed" as fake PoC. Standard: identity proven (name+email from dashboard) + victim data accessed (PII from authenticated endpoint).
- **TEST WRITE METHODS ON PROD UNAUTH ENDPOINTS (BlueSpider, June 2026):** When Phase 5/6 confirms GET endpoints work without auth, ALWAYS test POST/PUT/DELETE on action endpoints (reset-password, user-release, data-distribution) without ANY cookies. /api/reset-default-password accepted POST with zero auth on PROD — upgraded from "requires SA session" (original F-21) to "zero auth" (F-36 Critical). Endpoints that needed auth in prior engagements may lose auth requirements in code updates.
- **RE-ASSESSMENT PHASE DISCIPLINE (BlueSpider, June 2026):** When reopening phases, re-read the phase reference file and self-audit EVERY technique row. User caught gaps in ALL phases (1-7) by asking "have we done all activities properly?" - **INACTIVE USER ≠ EXPLOITABLE USER (BlueSpider, June 2026):** When testing password reset → login chains, ALWAYS check user_status first. INACTIVE accounts reject login even with correct password (302 back to /login). Pick an ACTIVE user for ATO proof. wasted attempts on mgr@aosmail.co.id (INACTIVE) before switching to meinetri.meinetri@agent.jago.com (ACTIVE).
- **POC MUST SHOW REAL DATA, NOT SUMMARIES (BlueSpider, June 2026):** PoC output must contain actual response data (real emails, real names, real phone numbers). NEVER show "166 records (129KB)" as PoC output — show the actual JSON records. A PoC that prints status codes and byte counts is NOT a PoC, it's a connectivity test. - **ATO PROOF REQUIRES AUTHENTICATED DATA ACCESS (BlueSpider, June 2026):** HTTP 204 or 302-to-dashboard does NOT prove ATO. You must: (1) login, (2) access an authenticated page (e.g., /dashboard), (3) extract the auth.user object or victim-specific data, (4) show victim's name/email/role in output. Without step 3-4, it's "login succeeded" not "account takeover confirmed." The 204 alone was rejected as "fake PoC" until output showed `Authenticated as: Meinetri (meinetri.meinetri@agent.jago.com)` + customer PII access.

Uphold (June 2026): full schema (9 mutations, all input types, enum values) reconstructed from error messages alone.
