## Guardrails

### Disclosure & Platform Rules

- **Public Disclosure Prohibition** — NEVER publish PoCs on public URLs before vendor fix (or 90-day deadline). Keep PoCs local in `ptest-output/report/`. See `references/bug-bounty-submission-guide.md`.
- **YesWeHack Dojo Challenges** — See `references/yeswehack-dojo-interaction.md` for UI interaction patterns.
- **IssueHunt Programs** — URLs use UUID format: `https://issuehunt.io/programs/{uuid}`. The SPA doesn't render via curl — use browser. Program pages require JS rendering to show scope/rewards/guidelines.

### Scope Rules

- **Scope Type Interpretation** — "Web application" (e.g., `mokapos.com`) means ONLY that domain, NOT `*.mokapos.com`. Only "Wildcard" targets include subdomains. Confirm BEFORE Phase 1.
- **IssueHunt scope** — IssueHunt lists specific domains in "In Scope" (e.g., "bitbank.cc, app.bitbank.cc, api.bitbank.cc"). Only those exact domains are reportable. Subdomains discovered via recon are informational for understanding infrastructure but findings on them are out-of-scope. Phase 2 DNS expansion and brute-force can be SKIPPED when scope is explicitly limited to named domains (not wildcard).
- **Related-Domain Scope Risk** — findings on same-company different-root-domain are borderline. Submit clear-scope findings first, borderline last with scope note.
- **Scope Enforcement** — never test targets outside defined scope. Re-read `scope.md` before each technique.
- **Program Exclusion Cross-Check (Phase 4, MANDATORY)** — cross-reference ALL vectors against the program's exclusion list BEFORE scoring them.

### Evidence Rules

- **Evidence Required** — every finding must have reproducible proof.
- **Verified Findings Only** — DNS resolution or CT log presence alone does NOT constitute a finding. Must include proof of current exploitability.
- **Environment Tagging** — every finding MUST be tagged: `prod`, `nonprod`, `experiment`, or `all`.

### Workflow Rules

- **IssueHunt program URLs are UUID-based** — don't guess from company name. Format: `https://issuehunt.io/programs/{uuid}`. The site is a JS SPA — curl won't render content, must use browser tools. Ask user for the URL if search engines are blocked.
- **HackerOne program slug discovery** — slugs are unpredictable (not always derived from company/product name). Try: exact name, hyphenated, underscored, parent company name. LINE Works → `line_works` (404), `lineworks` (404), `line` (correct — parent company LY Corp). When slug fails, search HackerOne directory or ask user. Also check program status BEFORE starting recon — suspended programs accept no submissions.
- **HMAC-auth APIs (not cookie-based)** — when the target uses custom header auth (ACCESS-KEY + HMAC signature), CORS misconfigurations without `Access-Control-Allow-Credentials: true` are NOT exploitable. CSRF is also not applicable. Don't waste time reporting these. Focus on: signature timing attacks, key/secret leakage in JS bundles, business logic, and auth flow weaknesses instead.
- **HTTP probing tiered policy (curl vs Python)** — use the right tool for the job:
  - **Tier 1 (curl OK):** Single-host connectivity checks (`curl -o /dev/null -w '%{http_code}'`), following redirects, checking response headers. Quick one-liners where quoting is trivial.
  - **Tier 2 (Python required):** Multi-host batch probing (5+ targets), injection/exploitation payloads (SQLi, SSTI, GraphQL), anything with special chars/JSON bodies, any request whose output becomes PoC evidence.
  - **Tier 3 (Python mandatory):** Final PoC scripts, anything submitted in reports, auth flows, session management testing.
  Rule of thumb: if you're writing a loop, escaping quotes, or the payload has `{}[]"'`, use Python. **User corrected (LINE WORKS, June 2026):** curl was used for complex GraphQL payloads causing quoting nightmares. The fix is tiered — not a blanket ban that gets ignored.
- **Test every asset individually** — do not assume subdomains behave the same because they share infrastructure. Each may have different WAF policies, backends, or exposed paths.
- **Never assume a host is "dead" based on surface appearance** — default pages (Apache2, nginx, IIS), empty-looking responses, or "abandoned" timestamps do NOT justify skipping mandatory enumeration tools. Run gobuster/feroxbuster on every live host. Manual spot-checks of a few paths are NOT a substitute for proper brute-forcing. Hidden apps live behind default pages. **Standard wordlists may not contain the paths** — if gobuster/ffuf/nuclei all return nothing, that does NOT mean the host is empty. Try: (1) context-specific paths derived from the org's naming patterns (app names, acronyms, project codes), (2) paths from other subdomains in the same org, (3) paths leaked in JS bundles or source code, (4) ask the user/client for known app paths. BFI lesson: e-pmo2.bfi.co.id showed Apache2 default page, gobuster/ffuf/nuclei all returned 0 findings, but 4 live PHP apps existed at /boarding, /room, /dfr, /kboard — paths no standard wordlist contains. **Do NOT declare a host dead after only running standard tools** — the user corrected: "don't be assume dude." Even when gobuster returns all-403, ffuf finds nothing after size filtering, and nuclei reports 0 findings, the host may still have live applications at paths you haven't guessed. Exhaust ALL discovery methods before concluding. **Additional discovery methods when standard tools fail:** (5) business-function guessing based on the subdomain name (e.g., "e-pmo" → project management → try /boarding, /room, /schedule, /report, /dashboard), (6) when all paths return identical responses (e.g., all 403 same size), that's WAF/server default-deny — it does NOT mean paths don't exist. The server returns 403 for BOTH non-existent AND existing-but-blocked paths. You need paths that return a DIFFERENT response (200, 302, different size) to discover real apps.
- **Phase 3 coverage = ALL live hosts from Phase 2** — not just the "interesting" ones. Get the unique live host count, categorize (accessible vs blocked), run techniques on ALL accessible hosts, document blocked ones with reason. Testing 13/54 hosts is NOT "Phase 3 complete." The same path (e.g., `/partner-webview/`) may be exposed on dev/stg/pt variants even when the root is blocked. "api.jago.com returns 401 so all *.api.jago.com must too" is a BAD assumption that was proven wrong.
- **Present ALL data first, then prioritize** — never cherry-pick "interesting" results and hide the rest. Show complete picture (all live hosts grouped by response code), then highlight priorities. User needs to see what was tested vs what wasn't.
- **Track tested vs status-checked** — "Probed" = path discovery, parameter fuzzing, tech-specific tests. "Status-checked" = one curl for HTTP code. Don't conflate them in reports.
- **When behind WAF/Access control** — don't give up after confirming the wall. Try: different paths, methods, UAs, HTTP versions, direct IP, alternate ports, websocket upgrade, path encoding. Use `xargs -P` with curl for fast parallel path discovery (gobuster often times out on rate-limited targets).
- **"Try harder" means** — full directory bruteforce (not 20 manual paths), every live subdomain gets path discovery, parameter fuzzing on every input-accepting endpoint, technology-specific CVE testing, multiple bypass techniques per WAF.
- **Findings don't excuse incomplete phases.** Discovering a high-value finding does NOT permit skipping remaining phase techniques or output documentation. Complete the phase checklist mechanically. LINE WORKS (June 2026): found GraphQL in Phase 6, got excited, skipped Phase 3 handoff, Phase 4 outputs, Phase 5 nuclei, and Phase 6 checklist. User caught every gap across 5 separate questions.
- **Strict Sequence & Self-Audit** — never skip a phase. Never skip steps within a phase. Complete ALL checklist items for the current phase before requesting gateway sign-off. The user explicitly corrected: "we will test all the scope dude, skip the step, then you may lost something." Before requesting sign-off, proactively review what was missed — list gaps honestly and offer to fill them. Do not suggest "pivoting to depth" or "skipping remaining Phase 1 items" — finish the phase completely first.
- **FULL COVERAGE RULE: Every phase, every subdomain, no exceptions.** Every technique in every phase runs against ALL live hosts — not a sample, not "key targets," not "interesting ones." Compare your tested-hosts list against the master subdomain list before requesting sign-off. Hosts behind WAF may have path-specific bypasses. User corrections across 4 engagements (Meta, BFI, Bank Jago, LINE WORKS): "did you do all the activities for all subdomains?" — the answer must always be YES before requesting gateway sign-off. If time is a concern, ask the user — don't silently skip.
- **Phase 4 Asset Inventory Coverage Verification (MANDATORY)** — after compiling the asset inventory, diff it against the full `live-subs.txt` (186 hosts in BFI). The original BFI Phase 4 only mapped 22/186 (12%) and missed critical findings: unauth APIs on SIT/UAT gateways, new Keycloak servers (sso.nonprod.bravo), CORS vulns. Count your inventory rows vs live-subs lines — if coverage is below 80%, you're not done. Hosts returning 404 at root may still have live services at specific paths (e.g., `/keycloak`, `/master/v1`, `/actuator`).
- **Phase 2→3 Coverage Verification (MANDATORY)** — before starting Phase 3, diff Phase 2 live-hosts.txt against Phase 1 subdomains-master.txt. Batch-probe ALL missing subdomains before enumeration. Quick batch probe: `python3 -c "import httpx; [print(f'{r.status_code} {len(r.content)} {host}') for host in open('missed.txt') for r in [httpx.get(f'https://{host.strip()}/', timeout=5, verify=False)]]"` or use `scripts/http-probe-parallel.sh`. Bank Jago (2026-05): 184 of 343 subdomains were missed in Phase 2, caught only when user asked "did we do phase 3 for all subdomains?"
- **Phase 1 OSINT Completeness** — verify ALL 10 techniques attempted: (1) WHOIS/DNS/TXT, (2) subdomain enum, (3) Wayback Machine, (4) GitHub/GitLab search, (5) Google dorking, (6) Shodan/Censys, (7) JS bundle analysis, (8) Mobile app identification, (9) Docker Hub check, (10) dark web & breach data OSINT.
- **Static-served WordPress CVE trap** — when nginx serves pre-rendered HTML (ETag + Last-Modified headers, no Set-Cookie on public pages), PHP-level CVEs (WP Fastest Cache SQLi via cookie, xmlrpc brute-force, etc.) are NOT exploitable because the vulnerable PHP code path is never invoked for unauthenticated requests. Verify by: (1) checking if response includes ETag/Last-Modified (= static file), (2) comparing response size with/without cookie (size diff = dynamic rendering activated), (3) testing SLEEP-based timing oracle (no delay = PHP not processing the payload). Don't waste time reporting "vulnerable version installed" without confirming the code path is reachable.
- **Phase 1 is STRICTLY passive** — no packets to target infrastructure. HTTP probing, port scanning, TLS cert grabbing (connecting to target), technology fingerprinting, and path discovery are ALL Phase 2 (Active Recon). Phase 1 uses only third-party data sources: DNS zone files, Shodan InternetDB, GitHub code search, Google dorks, CT logs, WHOIS, SPF/DMARC/TXT record analysis. If you're sending a SYN packet or HTTP request to the target, it's Phase 2.
- **Phase 2→3 handoff: probe ALL subdomains from master list** — see Phase 2→3 Coverage Verification above. Duplicate removed.
- **When source code reveals API endpoints, test EVERY one** — don't just analyze the source statically. Test each discovered endpoint against all relevant hosts with GET AND POST. Different HTTP methods may have different auth enforcement (GET blocked, POST allowed). Bank Jago (2026-05): Flutter main.dart.js revealed 52 API paths; testing POST on "forbidden" endpoints revealed CF API Shield bypass on /ginpay/* paths.
- **Mandatory Tool Execution** — mandatory tools per phase must be run. If unavailable, document the gap explicitly.
- **Human Sign-off** — always request user confirmation before passing a gateway.
- **No Time/Schedule Commentary** — never comment on the time or suggest stopping. The operator decides their schedule.
- **Complete Data First, Then Prioritize** — when presenting recon/scan results, show ALL data grouped by category (response codes, service types) before highlighting priorities. Never cherry-pick "interesting" results and omit the rest. The operator needs the full picture to make decisions. No assumptions about what they "already know".
- **Avoid Output Truncation** — when running batch probes or writing large inventory files via execute_code, keep individual outputs under 4KB. Split large writes into multiple patch() calls of ~2KB each rather than one massive write_file(). For terminal commands probing many hosts, batch in groups of 5-8 (not 15-20) and print results incrementally. The user explicitly corrected: "just be careful with the truncated problem dude" — twice. If output might exceed context limits: (1) write to file first, (2) print only a summary to stdout, (3) confirm with user before showing full details.
- **Authorization First** — refuse to begin without confirmed authorization.
- **No Deployed Persistence** — document persistence techniques but do not deploy backdoors without explicit authorization.
- **Never conclude Phase 6 prematurely** — if initial tests come back negative, that's the START of exploitation, not the end. Exhaust ALL vectors before declaring "no findings." The user corrected this (LINE WORKS, June 2026): agent concluded 0 findings and jumped to Phase 8 after surface-level tests. After reopening Phase 6 and trying harder (Elementor lightbox XSS, nginx case bypass, AASA deep link paths), a confirmed DOM XSS was found. **Minimum Phase 6 checklist before concluding:** (1) all known CVEs for identified versions tested with correct payloads, (2) nginx/WAF bypass techniques applied to all blocked paths, (3) all admin-ajax actions enumerated, (4) JS-level attack surfaces tested in browser (DOM XSS requires browser verification), (5) mobile app association files checked for hidden paths, (6) authentication flow paths probed.
- **ALWAYS do post-exploitation.** See `references/post-exploitation-rules.md`. Never stop at "proved access exists" — demonstrate actual impact.
- **NEVER conclude engagement prematurely.** When initial tests come back negative, that is NOT permission to skip remaining techniques and jump to reporting. User corrected (LINE WORKS, June 2026): Agent marked Phases 6-8 as PASSED with 0 findings after surface-level testing. When challenged ("why did you concluded that?"), agent went back, tried harder, and found CVE-2022-29455 DOM XSS — a valid Medium finding that was there all along. **Rule:** Exhaust ALL identified attack vectors before concluding. Specifically: (1) test every known CVE for the exact installed versions, (2) try all trigger mechanisms (click, hash, event dispatch), (3) check all content types in lightbox/modal handlers, (4) enumerate region/path prefixes systematically. If you think the target is "too hardened" — that's the signal to try harder, not to give up.
- **Scope Type Awareness** — skip techniques that don't apply to the engagement's scope type.
